The Seven Skills

Each skill has a distinct job in the pipeline. Listed in workflow order.

1
gigo
2
plan
3
execute
4
review

Plus: maintain (ongoing), snap (audit), eval (test)

gigo

Trigger: New project, no CLAUDE.md exists, “assemble”

Builds your expert team from scratch. Researches the best practitioners in the domain, identifies their specific philosophies, and blends 2-3+ real authorities per persona. Then scaffolds the project: CLAUDE.md, rules, references, snap audit system.

This is where it starts. Describe your project and GIGO does the research. Be as vague or specific as you want.

Pipeline role: Creates the assembled context that powers planning and review.

Example: “Building a CLI tool for database migrations. Go, targets Postgres and MySQL.” → GIGO proposes The Migration Architect (Andrew Kane + Sandi Metz), The CLI Designer (Steve Francia + Rob Pike), The Test Strategist (Mitchell Hashimoto).

gigo:maintain

Trigger: Need new expertise, want a checkup, project has grown, “maintain”

Ongoing maintenance for your expert team. Auto-detects severity and picks the right mode: targeted addition (fill a specific gap), health check (assess coverage and freshness), restructure (fix bloat and drift), or upgrade (modernize older setups with latest patterns).

Also invoked automatically by gigo:plan when it detects expertise gaps during planning.

Pipeline role: Keeps the team current as the project evolves.

gigo:plan

Trigger: You have a task, feature, or idea, “plan this”

Turns ideas into execution-ready plans. Full arc: explore project context → ask 2-3 clarifying questions → propose 2-3 approaches → design spec (with Conventions section) → write implementation plan with ordered tasks and dependencies.

Hard gate: no execution without an approved plan. The plan embeds team expertise as concrete requirements — this is where assembled context earns its keep (Phase 6: assembled caught partial unique index, SKIP LOCKED, pagination that bare missed).

Pipeline role: Assembled planning. Personas shape WHAT gets asked, not just formatting.

gigo:execute

Trigger: Plan exists and approved, “execute”

Runs approved plans using three tiers: agent teams (full parallelization with hook-enforced review), subagents (sequential with some parallelization), or inline (sequential, no isolation). Workers run bare — they get the spec with embedded conventions, not the personas.

Phase 7 proved this produces senior/staff-level code. The delta is in the spec, not the worker’s context.

Pipeline role: Bare execution. Workers produce best output with training alone + good spec.

gigo:review

Trigger: Task completed, code ready for review

Two-stage code review. Stage 1: spec compliance — “did you build the right thing?” Has the spec, checks implementation against requirements. Stage 2: engineering quality — “is the code good?” Focused on bugs, race conditions, maintainability.

Phase 8 proved two focused reviewers find more issues than one combined reviewer. Issues triaged as: auto-fix (worker handles), ask-operator (blocks), or accept (noted).

Pipeline role: Assembled review. Team catches what workers miss.

gigo:snap

Trigger: Session end, saving progress, “snap”

Session-end audit and protection protocol. Two jobs, in order: (1) Protect the project — 10-point audit checking line caps, derivability, overlap, staleness, cost, persona calibration, total budget, coverage, overwatch, and pipeline integrity. (2) Capture learnings — route new knowledge to the correct file.

“Whatever it takes to protect the project.” A lean project with missing knowledge outperforms a bloated one with everything.

Pipeline role: Session boundary. Prevents the monotonic growth that degrades performance over time.

gigo:eval

Trigger: Want to prove context actually helps, “eval”

Tests whether assembled context actually improves output. Two modes: pipeline eval (default) runs multi-stage chains with different context levels and comparative judging. Isolated eval runs quick A/B checks on single tasks.

Methodology: same judge sees all variants, randomized labels, evidence-required scoring. “Evidence over vibes.”

Pipeline role: Measurement. Without evals, context engineering is guesswork.
