gigo

Trigger: new project, no CLAUDE.md exists

Builds your expert team from scratch. Researches domain authorities, blends their philosophies into focused personas (2-3+ real practitioners per persona), and scaffolds lean project files. This is where it all starts — describe your project and GIGO does the research.

Creates CLAUDE.md (team roster), .claude/rules/ (standards, workflow), and .claude/references/ (deep knowledge, loaded on demand).

You: "Building a CLI tool for database migrations. Go, targets Postgres and MySQL."

GIGO proposes: The Migration Architect (Andrew Kane + Sandi Metz), The CLI Designer (Steve Francia + Rob Pike), The Test Strategist (Mitchell Hashimoto). You refine, lock in, and the team is assembled.

gigo:maintain

Trigger: need new expertise, want a checkup, project has grown

Ongoing team maintenance. Four modes: targeted addition (fill a specific expertise gap), health check (assess coverage and freshness), restructure (fix bloat and structural drift), upgrade (modernize older setups with latest patterns). Auto-detects severity.

Invoked automatically by gigo:plan when it detects expertise gaps during planning.

gigo:plan

Trigger: you have a task, feature, or idea that needs planning

Full planning arc: explore project context, ask 2-3 clarifying questions, propose 2-3 approaches, design, write spec (with a Conventions section embedding team standards), write implementation plan with ordered tasks and dependencies.

Hard gate: no execution without an approved plan. The spec embeds the team's expertise as concrete requirements — not rules for the worker to comply with, but decisions already made by experts who thought about the hard problems.

gigo:execute

Trigger: plan exists and is approved

Runs approved plans. Three tiers: agent teams (full parallelization with hook-enforced review), subagents (sequential with some parallelization), inline (sequential, no isolation). Detects available tools and picks the best tier.

Key principle: workers run bare. They get the spec with embedded conventions, not the personas. Phase 7 proved this produces senior/staff-level code — the delta is in the spec quality, not the worker's context.

gigo:review

Trigger: task completed, code ready for review

Two-stage code review. Stage 1: spec compliance — "did you build the right thing?" Checks implementation against plan requirements. Stage 2: engineering quality — "is the code good?" Checks for bugs, race conditions, test quality, maintainability.

Phase 8 proved two focused reviewers find more issues than one combined reviewer. Issues are triaged: auto-fix (worker handles), ask-operator (blocks for human decision), accept (noted but not blocking).

gigo:snap

Trigger: wrapping up, saving progress, end of session

Session-end audit. Two jobs in order: (1) protect the project — a 10-point audit checking line caps, derivability, overlap, staleness, cost, persona calibration, total budget, coverage, overwatch presence, and pipeline integrity. (2) Capture learnings — route new knowledge to the correct file.

"Whatever it takes to protect the project." Job 1 is more important. A lean project with missing knowledge outperforms a bloated one with everything.

gigo:eval

Trigger: want to prove assembled context actually improves output

Context effectiveness testing. Pipeline eval (default): multi-stage chains with different context levels, comparative judging. Isolated eval: quick A/B checks on single tasks. Methodology: same judge sees all variants, randomized labels, evidence-required scoring.

"Evidence over vibes." Use this when you want to prove your setup works or debug why it doesn't.
