Architecture

The Problem

Context files grow monotonically. Every session discovers something worth remembering, so rules files grow — adding, never removing. Within weeks: 200+ lines of overlapping rules, outdated guidance, reference-depth detail loading on every conversation.

Performance degrades. Sessions take longer, make more mistakes, sometimes ignore rules entirely. There’s too much to attend to.

Then Gloaguen et al. (2026) published the first rigorous study confirming it: bloated context files reduce task success rates while increasing cost by 20%+. You pay more and get worse results.

Two Tiers

The key innovation: “When to Go Deeper” pointers. Rules files don’t just point to references — they tell the agent when to read them. Not generic “see also” links but task-specific triggers:

“When tuning cooperative balance, read .claude/references/cooperative-balance.md.”

This makes the system task-aware: deep context loads only when relevant. Phase 2a proved it — task-specific pointers took the children’s novel from 86% to 100%.

The ~60 Line Cap

Not arbitrary. Research-informed. A 30-line file that nails the essentials outperforms a 100-line file that’s thorough but dilutes attention.

Approaching ~60 lines triggers a move: keep the essential rule in rules/, move the depth to references/. Total budget across all rules files: ~300 lines.

The Non-Derivable Rule

Only write what Claude can’t figure out by reading the project files.

Write these — philosophy, quality bars, anti-patterns, blended authorities. They’re not discoverable from the code.

Never write these — directory structure, code patterns, file organization. The agent navigates codebases on its own. Codebase overviews are actively harmful: they waste attention on information the agent would have found anyway.

The Snap

Named after Tony’s snap, not Thanos’s. Not indiscriminate wiping — sacrificing what has to go so what matters survives.

The 10-point audit runs every session end:

  1. Line check — any file approaching ~60 lines?
  2. Derivability check — can the agent figure this out from the code?
  3. Overlap check — rules saying the same thing in different words?
  4. Staleness check — rules that no longer apply?
  5. Cost check — worth loading on every conversation?
  6. Persona calibration check — domain knowledge in rules that belongs in references?
  7. Total budget check — approaching ~300 lines across all files?
  8. Coverage check — project outgrown the team?
  9. Overwatch check — adversarial self-check system present?
  10. Pipeline check — three phases (plan, execute, review) still intact?

Projects get sharper over time, not bigger.

Persona Architecture

Blended philosophies: 2-3+ real authorities per persona, each bringing a specific contribution. Not “a senior developer” but “Andrew Kane’s database ops pragmatism + Sandi Metz’s ‘small objects that talk to each other’ + DHH’s convention-over-configuration.”

Alignment signal (quality bars, approach, constraints) stays in rules. Knowledge signal (factual specifics, implementation patterns) goes in references.

The Hu et al. tradeoff: personas help alignment but hurt knowledge retrieval. Addressed by the persona calibration heuristic — tells the model when to lead with training vs persona. Phase 2a: this added 9 percentage points.

The Pipeline

Plan (assembled)
→
Execute (bare)
→
Review (assembled)

The team’s expertise shapes questions during planning. Phase 6: assembled catches partial unique index, pagination, copy condition that bare misses. Expertise gets embedded as requirements in the spec.

Bare workers follow the spec and produce senior/staff-level code. Phase 7: bare rated senior or staff every time by a principal engineer review.

The team reviews output against quality bars. Phase 8: two focused reviewers (spec compliance + engineering quality) find more issues than one combined reviewer.
