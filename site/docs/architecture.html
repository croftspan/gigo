Architecture

The Problem

Context files grow monotonically. Every session discovers something worth remembering, so rules files grow — adding, never removing. Within weeks: 200+ lines of overlapping rules, outdated guidance, reference-depth detail loading on every conversation.

Performance degrades. Sessions take longer, make more mistakes, sometimes ignore rules entirely — there's too much to attend to.

Gloaguen et al. (2026) published the first rigorous study confirming what we'd experienced: bloated context reduces task success rates while increasing cost by 20%+. Irrelevant rules dilute the model's focus on rules that actually matter. More requirements can make tasks harder, not easier.

Two Tiers

Location.claude/rules/.claude/references/

The key innovation: "When to Go Deeper" pointers in rules files tell the agent when to read a reference. Not generic "see also" links — task-specific triggers:

"When tuning cooperative balance, read .claude/references/cooperative-balance.md."

This makes the system task-aware. The agent loads deep context only when it's relevant to the current work.

The ~60 Line Cap

Not arbitrary. Research-informed.

A 30-line file that nails the essentials outperforms a 100-line file that's thorough but dilutes attention. When a rules file approaches ~60 lines, it triggers a move: keep the essential rule in rules/, move the depth to references/.

Total budget across all rules files: ~300 lines. Fewer is better.

The Non-Derivable Rule

Only write what Claude can't figure out by reading the project files.

Write: Philosophy, quality bars, anti-patterns, blended authorities — these aren't discoverable from the code.

Don't write: Directory structure, code patterns, file organization — the agent navigates codebases on its own. Codebase overviews are actively harmful because they waste attention on information the agent would have found anyway.

The Snap

Named after Tony's snap, not Thanos's. Not indiscriminate wiping — sacrificing what has to go so what matters survives.

A 10-point audit runs every session end:

  1. Line check — any rules file approaching ~60 lines?
  2. Derivability check — can the agent figure this out from the code?
  3. Overlap check — rules saying the same thing in different words?
  4. Staleness check — early-project rules that no longer apply?
  5. Cost check — worth loading on every conversation?
  6. Persona calibration check — knowledge content in personas that belongs in references?
  7. Total budget check — approaching ~300 lines across all rules?
  8. Coverage check — has the project grown into areas the team doesn't cover?
  9. Overwatch check — adversarial self-check system present?
  10. Pipeline check — three phases (plan, execute, review) still intact?

Projects get sharper over time, not bigger.

Persona Architecture

Blended philosophies: 2-3+ real authorities per persona, each bringing a specific contribution. Not "a senior developer" — "Andrew Kane's database ops pragmatism + Sandi Metz's 'small objects that talk to each other' + DHH's convention-over-configuration."

Alignment signal (quality bars, approach, constraints) stays in rules. Knowledge signal (factual specifics, implementation patterns) goes in references.

The Hu et al. tradeoff: personas help alignment but hurt knowledge retrieval. Addressed by the persona calibration heuristic — 6 lines that tell the model when to lead with training vs lean into persona. On presentation tasks, persona leads. On content tasks, training leads, persona frames.

The Pipeline

Plan (assembled)
→
Execute (bare)
→
Review (assembled)

Planning: The team's expertise shapes questions. Phase 6 showed assembled catches partial unique index, pagination, copy condition — architectural improvements bare misses. Expertise gets embedded as requirements in the spec.

Execution: Bare workers follow the spec and produce senior/staff-level code. Phase 7 proved the format of execution context doesn't matter — the spec quality determines output.

Review: Two focused reviewers: spec compliance ("did you build the right thing?") then engineering quality ("is the code good?"). Phase 8 proved two focused reviewers find more issues than one combined reviewer.
