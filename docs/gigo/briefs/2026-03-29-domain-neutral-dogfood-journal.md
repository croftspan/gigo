# Dogfood Journal: Domain-Neutral Review Templates

How we used GIGO to make GIGO work for everyone.

---

## The Problem

GIGO claims to work for any domain. Software, children's novels, tabletop board games, research papers. But the review system assumed code. The Challenger asked about "race conditions." The spec reviewer wanted "file:line references." A children's novel project would get reviewed for "deadlocks."

The operator caught it during a live pipeline run: "Our reviews and sanity checks need to be domain-agnostic. It's not always code, sometimes it's children's novels, sometimes it's tabletop board games."

## What We Did

We used `/gigo:blueprint` to plan, spec, and execute the fix. The pipeline that reviews code was used to redesign how it reviews anything.

### Plan Mode (Phase 0-4)

Blueprint entered plan mode and explored the existing review templates. Found 40+ code-specific terms across three templates. Designed the progressive enrichment pattern: neutral defaults with a `{DOMAIN_CRITERIA}` injection point for project-specific checks.

The fact-checker (Phase 4.25) caught two issues before the design brief was even approved:
- verify SKILL.md hardcodes the same section names being changed in the templates
- Step 6.5 is an append after the final gigo step, not an insert between steps

### The Spec

The spec defined exact replacements for every code-specific term:
- "Race conditions, deadlocks, lock ordering" became "Look for defects the author likely did not intend"
- "file:line references" became "specific location references"
- "engineering quality" became "craft quality"
- "code changes" became "changes"
- "codebase" became "project"

The Conventions section locked in `{DOMAIN_CRITERIA}` as the universal variable name across all three templates.

### The Challenger Finds

**Spec Challenger (4/5, Proceed):**
- Step 6.5 insertion point clarification
- PR mode won't get domain criteria (documented as known limitation)
- The classification step requires judgment, not purely mechanical
- Snap step 11 pushes snap.md to 57 lines (near cap but acceptable)

**Plan Challenger (4/5, Proceed with revisions):**
- Dangling reference to old `engineering-reviewer-prompt.md` in verify SKILL.md Pointers section
- Phase 9.5 had lowercase "quality bar checklist" and "code in task steps" that would have survived the rename

Both catches were real. The Pointers reference would have been a broken link. The Phase 9.5 language would have been the only code-specific text left in the pipeline.

### Execution

11 tasks across 4 phases. The plan had Tasks 1-3 as parallelizable (template neutralization), Tasks 4-5 as dispatcher updates, Tasks 6-9 as generation machinery, Tasks 10-11 as eval fixture generation.

We asked for subagent execution. It ran inline. That's a known issue we haven't solved yet. But the work was correct.

### The Result

Three review templates now work for any domain. The `{DOMAIN_CRITERIA}` variable gets filled from `.claude/references/review-criteria.md` when it exists, stays empty when it doesn't. A Go project gets "Does every goroutine have a shutdown path?" A children's novel gets "Is every clue planted 2+ chapters before payoff?" A board game gets "Does the reward loop respect the session-length target?"

The criteria are generated automatically by `gigo:gigo` as the last step of team assembly. They're regenerated by `gigo:maintain` when the team changes. The Snap audits them for staleness.

## Why This Matters

### The adversarial system is a pillar

The Challenger found real issues in its own redesign spec. A runtime blocker (Plan subagent constraint) was caught during the fact-checker run. Dangling references were caught during the plan review. These aren't theoretical catches. They're bugs that would have shipped.

Nobody else offers this. An independent reviewer that runs blind-first, can't rationalize the planner's choices, gives honest 3-4/5 scores instead of glazing, and works for children's novels as well as Go microservices.

### This is gatekept knowledge made accessible

The review quality that GIGO provides is what you get from a principal engineer review. Someone who reads your spec, checks it against the actual project, finds the things you missed, and gives you honest feedback. That expertise is normally gatekept behind years of experience and knowing the right people.

GIGO generates it from the team's quality bars. A children's book author gets the same rigor as a senior engineer. Different criteria, same discipline.

### The dogfood proves it works

We didn't theorize about domain-agnosticism. We used the pipeline to make itself domain-agnostic. The same Challenger that reviews your specs reviewed its own redesign spec. The same fact-checker that checks your design briefs caught issues in its own design brief.

Every improvement to GIGO's review system was reviewed by GIGO's review system. That's not a demo. That's production use.
