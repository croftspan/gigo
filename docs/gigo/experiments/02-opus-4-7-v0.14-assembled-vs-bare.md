## The question

Does the GIGO team-in-context still meaningfully outperform bare Claude on Opus 4.7, after v0.13 (Two-Gate research pipeline) and v0.14 (mission-control integration) landed?

Neither v0.13 nor v0.14 touched the fixtures themselves — both were pipeline-internal changes. The test isolates a single new variable: the model. The fixtures are preserved at their pre-v0.13 SHAs to keep the comparison apples-to-apples with the 2026-04-12 baseline.

## Harness fixes applied

Two pre-flight fixes landed on `main` before the run:

1. **Pin the model** (commit `931874f`) — `run-eval.sh` was invoking `claude -p` with no `--model` flag, which meant "whichever CLI default the host happens to have." Added `--model claude-opus-4-7` to both bare and assembled invocations. Result names now anchor to a specific model ID.
2. **Bare-contamination glob** (commit `9e6ad3c`) — the fixture directories carry `CLAUDE.md.original` and `CLAUDE.md.firstperson` alongside `CLAUDE.md`. The old exact-match exclusion (`[ "$fname" != "CLAUDE.md" ]`) let the variants leak into the "bare" temp dir, silently inflating bare-condition quality. Switched to a case statement skipping the full `CLAUDE.md*` glob. Verified post-fix: bare tmpdir contains only the domain source files (Gemfile, app/, config/, db/ for rails-api; chapters/, characters.md, plot-outline.md for childrens-novel).

Both fixes are load-bearing for comparability with prior and future runs.

## Observations

### 1. Opus 4.7 has notably strong cold-context pushback behavior

The most striking finding from the bare responses: given an ambiguous short instruction in a low-context working directory, Opus 4.7 frequently **hallucinates structure in the input** to justify refusing to act. Across both domains, 6+ of the 20 bare responses opened by interpreting the prompt as "a menu of possible prompts (grouped A/B/C)" or "a classification exercise" — even though only the command portion of each prompt line was ever passed to the model. A few examples (quoted from the results):

- rails-api #01: *"Your message looks like a menu of possible prompts (grouped A/B/C) rather than a specific request. I don't see a working directory..."* (cwd actually had `app/`, `config/`, `db/`, `Gemfile`)
- rails-api #06: *"I don't see a controller in your message — just the prompt text itself with some categorized options below it."* (no categorized options were sent)
- childrens-novel #04: *"This looks like a multiple-choice quiz rather than a coding task. The answer is A|..."* (answering its own hallucinated quiz)
- childrens-novel #09: *"C"* (single-letter response to "Write chapter 7")

The model is pattern-matching defensively — fabricating justification to push back on grounding it can't confirm. The assembled condition doesn't fix the grounding problem by giving the model more cwd-exploration instructions; it fixes it by giving the model a **stance** (the persona) and a **set of rules to cite** when pushing back. Same prompt, same cwd, same model — but the assembled response picks up the task and runs with it.

This behavior appears to be newer/stronger in Opus 4.7 than in the models that produced the 2026-04-12 baseline. It's worth testing against Sonnet 4.6 and Haiku 4.5 in a future eval to see whether the cold-context-pushback effect is model-family-wide or specific to Opus.

### 2. One judge-rubric caveat: prompt rails-api #06

The judge awarded assembled a 5/5 sweep on prompt 06 ("Review this controller"), but the judge's own note reads:

> *"A honestly asked for the missing code while B hallucinated a controller and reviewed the fabrication — but B's review, taken on its own terms, demonstrates far richer domain-specific judgment, persona voice, and rule citation."*

That is: **the assembled response fabricated a controller and then produced a rich review of the fabrication**, while the bare response honestly asked for the controller code. The rubric rewarded the fabrication because it looked more thorough.

This is a genuine weakness in the current judge rubric. The rubric measures quality-of-review-taken-on-its-own-terms without discounting for hallucinated substrate. In a real use-case, assembled's response would be actively harmful — generating confident critique of code the user never wrote. Flagged for future rubric work; for this eval the verdict stands as the rubric scored it.

### 3. Verbosity is not the explanation

One common skepticism of assembled-vs-bare evals is "assembled just wins because it writes more tokens and judges conflate length with quality." The numbers:

- Average bare response: 1,027 output tokens
- Average assembled response: 2,701 output tokens (2.6×)

But the judge notes consistently reward *specificity* (named personas, named files, named anti-patterns), not length. Prompt 09 rails-api is instructive: bare's response was substantive and Beck-like in spirit ("figure out why they're slow before optimizing") — it just didn't cite `standards.md` or the factory-design-order rule by name. That's where the specificity-criterion win came from, not from the word count.

### 4. The rails-api gap closed almost entirely

The 2026-04-12 baseline had rails-api at 66% while childrens-novel sat at 98-100%. The hypothesis for the rails-api floor was that Rails domain expertise is *common* in Claude's training — a well-resourced bare response can speak credibly about migrations, pagination, or deploy gates without needing a team persona to point at. Children's-novel craft rules (fair-play mystery conventions, Blume's bar, plant-and-payoff structure) are rarer in training and harder to simulate bare.

The 2026-04-19 run collapses that gap to 98% vs 100%. Two non-exclusive reads:

- (a) Opus 4.7's cold-context pushback is strong enough to tank bare's rails-api showing — the model refuses to engage rather than produce generic-but-competent Rails advice, which was the bare win source before.
- (b) The v0.14 plugin, while it didn't change the fixtures, *does* load its pipeline rules — `workflow.md`, `standards.md`, the Overwatch checklist — into every assembled invocation. Those files have more explicit rule-citing language now than they did at v0.11.

Disentangling (a) and (b) requires running v0.11 fixture CLAUDE.md files against Opus 4.7 — a follow-up run.

## Recommendation

**v0.14.0-beta + Opus 4.7 clears the 90% assembled-win threshold by a comfortable margin (99% vs 90%). No Phase 2 intervention indicated from this run.**

The honest caveats:

1. **Watch the cold-context-pushback effect.** If Opus 4.7's tendency to refuse ambiguous prompts in empty contexts is artificially widening the assembled/bare gap, the "assembled earns its tokens" story becomes partly a story about "Opus 4.7 needs context scaffolding to commit to an answer." That's still a win for assembled context — but it's a different claim than "assembled context improves quality."
2. **Fix the judge rubric's hallucination-credit blind spot.** Prompt 06 shows the rubric currently rewards confident-but-fabricated reviews over honest-about-missing-inputs pushback. Future judge versions should penalize hallucinated-substrate reviews explicitly.
3. **The rails-api floor closed — verify it's not Opus 4.7 flooring itself.** Run the same harness against Sonnet 4.6 with the same fixtures to measure whether the 66% → 98% jump is "v0.14 fixed rails-api" or "Opus 4.7 floors without context."
