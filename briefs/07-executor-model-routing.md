# Brief: Executor Model Routing

## The Problem

`gigo:execute` dispatches work through Claude's Agent tool — every task spawns a subagent running on Claude. This works but costs Opus tokens for execution work that a local model handles just as well. The Gemma eval proved: on spec-following tasks, Gemma 4 at 26B-A4B hits 100% EXECUTES in 16 seconds. Opus tokens should be spent on planning and review, not execution.

## What We Know

From the Gemma eval and architecture analysis:

- **26B-A4B for speed** — 16s/prompt at temp 1.0, 100% EXECUTES. 6.7x faster than 31B. Best for straightforward spec execution.
- **31B for judgment** — 107s/prompt, 100% EXECUTES, holds pushback rules. Better when the executor needs to override poorly-formed instructions.
- **llama-server exposes OpenAI-compatible API** — `/v1/chat/completions` at localhost:8080. Health check at `/health`. Model detection at `/v1/models`.
- **The harness is the system prompt** — the Gemma-compatible context (from brief 06) gets passed as the `system` message. The task instruction is the `user` message.
- **Workers run bare** — this is already GIGO's model. Workers get the task spec, not the full assembly. This maps perfectly to Gemma execution.

## What to Build

### Model Detection

When `gigo:execute` starts a task, check for a local model:

```
GET http://localhost:8080/health → 200 means local model available
GET http://localhost:8080/v1/models → returns model ID (determines 26B vs 31B)
```

If available: route execution tasks to local model.
If not: fall back to Claude Agent subagents (current behavior).

### Execution Path

For local model execution:

1. Load the Gemma harness (generated by brief 06, or hand-tuned)
2. Format the task as a user prompt — the plan's task description + relevant context files
3. POST to `/v1/chat/completions` with harness as system, task as user
4. Parse response — extract code blocks, file paths, validation results
5. Apply the code (write files, run commands) — this part needs thought
6. Return result to the execution pipeline for review

### The Hard Part: Step 5

Claude Agent subagents can read files, write files, run tests, iterate. A llama-server completion is just text — code blocks in a response. Something needs to:
- Parse the code blocks and their file path headers
- Write the files to disk
- Run any commands (migrations, tests)
- Handle failures (re-prompt Gemma with the error, or escalate to Claude)

Options:
- **A: Thin shim in gigo:execute** — parse response, write files, run tests in bash. Simple but brittle.
- **B: Hermes agent loop** — Gemma runs inside an agent framework that handles tool use. More robust but bigger build. This connects to brief 09.
- **C: Hybrid** — Gemma generates the code, Claude applies it. Opus tokens only for file I/O and test running, not for thinking about what to write.

### Testing Required

**Integration test:**
- Take a plan with 3-5 tasks from a previous `gigo:execute` run
- Execute via local model routing
- Compare output quality to the Claude subagent execution
- Check: files written correctly, tests pass, spec compliance

**AB test design:**
- Same task, Claude subagent vs Gemma local
- Measure: time, token cost, spec compliance (via gigo:verify), code quality
- Run on Rails API fixture tasks where we have baselines

**Routing correctness:**
- Local model unavailable → falls back to Claude (no error)
- Local model returns garbage → escalates to Claude (not silent failure)
- Model detection identifies 26B vs 31B correctly

**What could go wrong:**
- Response parsing fails on edge cases (nested code blocks, non-standard headers)
- Gemma generates code that references files it hasn't read (no tool use = no file reading)
- Multi-step tasks that need intermediate results between steps
- llama-server crashes mid-generation on large responses

## Dependencies

- Should happen AFTER brief 06, BEFORE brief 09
