# tq add + tq list — Design Spec

## Goal

Add two commands to tq: `tq add` for enqueuing tasks and `tq list` for displaying them. Both use the existing `Store` interface and are independent of each other — they can be built in parallel by separate workers.

Also: a review hook (`.claude/hooks/gigo-review-gate.sh`) for automated two-stage review during `gigo:execute`.

## Architecture

Same three-package structure from the status spec. Two new files in `cmd/`, no changes to `store/` or existing files.

```
cmd/
├── root.go          # existing — root command, --json flag
├── status.go        # existing — unchanged
├── status_test.go   # existing — unchanged
├── add.go           # NEW — tq add command
├── add_test.go      # NEW — tests for add
├── list.go          # NEW — tq list command
└── list_test.go     # NEW — tests for list
.claude/
└── hooks/
    └── gigo-review-gate.sh  # NEW — review hook
```

No changes to `store/store.go` or `store/memory.go`. Both commands use `Store.Add` and `Store.List` as-is.

## Command: `tq add`

### Usage

```
tq add <name> --cmd "shell command" [--priority N] [--depends-on id1,id2]
```

### Arguments and Flags

| Argument/Flag | Required | Default | Description |
|---|---|---|---|
| `<name>` | yes | — | Positional arg. Task name. Exactly one. |
| `--cmd` | yes | — | Shell command the task runs. String. |
| `--priority` | no | `0` | Integer. Higher = more important. |
| `--depends-on` | no | `""` | Comma-separated task IDs. No spaces. |

### Behavior

1. Validate: exactly one positional arg, `--cmd` is non-empty
2. If `--depends-on` provided, parse comma-separated IDs into a `[]string`
3. If `--depends-on` provided, call `store.List(Filter{})` and verify each dep ID exists. Error if any missing.
4. Determine state: `StateReady` if no dependencies, `StatePending` if dependencies exist
5. Build `store.Task{Name: name, State: state, Priority: priority, DependsOn: deps, CreatedAt: time.Now()}`
6. Call `store.Add(task)` — ID is auto-generated by the store
7. Retrieve the generated ID from the store (call `store.List` and find the task by name+createdAt, or modify the approach — see Implementation Note below)

### Implementation Note: Getting the Generated ID

The current `Store.Add` doesn't return the generated ID. Two options:

**Option A (recommended):** Pre-generate the ID in the command, set it on the Task before calling `Add`. Use the same `crypto/rand` + `hex.EncodeToString` pattern from MemoryStore. This avoids needing to query back.

**Option B:** Modify the Store interface to return the ID. This changes an existing interface — unnecessary for this feature.

Use Option A. The `add` command generates its own ID and passes it to `Store.Add`. The store's auto-generation still works for other callers that leave ID empty.

### Output

**Default (text):** Print only the task ID to stdout. Nothing else.

```
a1b2c3d4
```

**JSON (`--json`):**

```json
{"id":"a1b2c3d4"}
```

**Errors:** To stderr, exit 1.

```
tq: cannot add task "deploy": --cmd is required
tq: cannot add task "deploy": dependency "xyz" not found
```

### Testing

Use `runAddWithStore(s store.Store, name string, cmdStr string, priority int, dependsOn []string, jsonOut bool, w io.Writer) error` — same injectable pattern as `runStatusWithStore`.

Tests:
- Add with all flags → verify task appears in store with correct fields
- Add without `--cmd` → error
- Add with no positional arg → error (Cobra handles this via `Args: cobra.ExactArgs(1)`)
- Add with `--depends-on` referencing nonexistent ID → error
- Add with valid `--depends-on` → state is `StatePending`
- Add without `--depends-on` → state is `StateReady`
- Add with `--json` → output is `{"id":"..."}` and parseable
- Add default output → output is just the hex ID, no newline issues

## Command: `tq list`

### Usage

```
tq list [--state ready|pending|...] [--json]
```

### Flags

| Flag | Required | Default | Description |
|---|---|---|---|
| `--state` | no | all | Filter by state. One of: pending, ready, running, done, failed, dead. |
| `--json` | no | `false` | Inherited from root. JSON array output. |

### Behavior

1. If `--state` provided, validate it's a known state. Error if not.
2. Build `store.Filter{State: &state}` (or `Filter{}` for all)
3. Call `store.List(filter)`
4. Sort results: priority descending, then CreatedAt ascending (oldest first as tiebreaker)
5. Format and print

### Output

**Default (text):** Header row + one row per task. Matches the format from `cli-patterns.md`.

```
ID        STATE     PRI  NAME            DEPENDS
a1b2c3d4  ready      10  build-frontend
e5f6a7b8  pending     5  deploy          a1b2c3d4
```

Column widths: `%-10s%-10s%4d  %-16s%s`. The DEPENDS column is comma-separated dep IDs, or empty.

Empty list: no output at all, exit 0. Not an error.

**JSON (`--json`):**

```json
[
  {"id":"a1b2c3d4","name":"build-frontend","cmd":"npm run build","state":"ready","priority":10,"depends_on":[],"created_at":"2026-03-27T10:00:00Z"},
  {"id":"e5f6a7b8","name":"deploy","cmd":"./deploy.sh","state":"pending","priority":5,"depends_on":["a1b2c3d4"],"created_at":"2026-03-27T10:01:00Z"}
]
```

Empty list: `[]` (empty JSON array), exit 0.

Note: JSON uses `depends_on` (snake_case) and `created_at` for consistency with JSON conventions. JSON output includes `cmd` even though the text table does not — machines benefit from the full record.

**Error — invalid state:**

```
tq: invalid state "borked" (valid: pending, ready, running, done, failed, dead)
```

### JSON Struct

```go
type ListTask struct {
    ID        string   `json:"id"`
    Name      string   `json:"name"`
    Cmd       string   `json:"cmd"`
    State     string   `json:"state"`
    Priority  int      `json:"priority"`
    DependsOn []string `json:"depends_on"`
    CreatedAt string   `json:"created_at"`
}
```

`DependsOn` is always an array (empty `[]`, never `null`). `CreatedAt` is RFC 3339.

### Testing

Use `runListWithStore(s store.Store, state string, jsonOut bool, w io.Writer) error`.

Tests:
- List empty store → no output, no error
- List with tasks → verify header + rows, correct format
- List with `--state ready` → only ready tasks
- List with invalid `--state` → error with valid options
- List with `--json` → parseable JSON array with correct fields
- List `--json` empty → `[]`
- List sorting → higher priority first, older tasks first within same priority

## Review Hook

### File: `.claude/hooks/gigo-review-gate.sh`

This is the hook script that `gigo:execute` invokes for Tier 1 automated review after each task completion. It runs two stages:

1. **Go quality gate:** `go vet ./...` and `go test ./...` — fast, catches obvious problems
2. **Two-stage review:** Invokes `gigo:verify` for spec compliance then engineering quality

The hook is a shell script, not a Go file. It exits 0 on pass, non-zero on failure.

```bash
#!/usr/bin/env bash
set -euo pipefail

# Stage 1: Go quality gate
echo "=== Stage 1: Go quality gate ==="
go vet ./...
go test ./...

# Stage 2: Two-stage review (spec compliance + engineering quality)
echo "=== Stage 2: gigo:verify ==="
claude -p "Run gigo:verify on the most recent changes. Two stages: spec compliance first, then engineering quality. Spec: docs/gigo/specs/2026-03-27-tq-add-list-design.md"
```

The hook assumes `claude` CLI is available (it runs inside a Claude Code session). The `gigo:verify` invocation handles both stages internally.

## Conventions

These apply to both new commands, extending the conventions from the status spec:

- **Error messages:** `tq: cannot <verb> task "<name>": <reason>` — always `tq:` prefix, verb matches command
- **Output:** Only essential data to stdout. Errors to stderr. `add` prints only the ID. `list` prints the table or JSON array.
- **Commands:** The `--cmd` flag value is stored as-is on the Task. Execution will wrap it as `["sh", "-c", <cmd>]` later — the add command just stores the string.
- **Exit codes:** 0 for success (including empty list), 1 for errors. Cobra handles usage errors (exit 2).
- **State naming:** `State` not `Status`, constants prefixed `State` (StatePending, StateReady)
- **Flag naming:** `--cmd`, `--priority`, `--depends-on`, `--state`. Kebab-case. No short flags for now.
- **No ANSI:** No colors, no formatting that assumes a terminal.
- **JSON struct:** Named structs with `json` tags. Never `map[string]interface{}`.
- **Store access:** Commands get a `Store`. For now, create `MemoryStore` in `RunE`. Testable via injectable `runXWithStore` functions.
- **Errors from RunE:** Return `fmt.Errorf(...)`, never call `os.Exit` in command logic.
- **Package boundaries:** `cmd/` imports `store/`. `store/` imports nothing from `cmd/`.
- **ID generation:** 8-char hex from `crypto/rand` (4 random bytes). Same pattern as MemoryStore.
- **Null safety in JSON:** `DependsOn` must serialize as `[]` not `null`. Initialize empty slices.
- **Sorting:** `list` sorts by priority descending, then CreatedAt ascending. `sort.Slice` in the command, not in the store.

## Task: Storing the Command String

The `Task` struct currently has no `Cmd` field. The `add` command needs to store the shell command string.

**Decision:** Add a `Cmd string` field to `store.Task`. This is the one store change needed. Both workers need it — Worker 1 (`add`) writes it, Worker 2 (`list`) displays it (or not — it's not in the list output columns currently). Since it's a single-line struct field addition with no behavior change, include it in both task specs as a prerequisite step. No merge conflict risk — both workers add the same line to the same struct.

Actually, looking at the `list` output format from `cli-patterns.md`, the columns are `ID STATE PRI NAME DEPENDS` — no CMD column. And `add` just needs to store it. So the `Cmd` field is written by `add` and read later by the worker pool (not yet built). For now, it's stored but not displayed by `list`.

**Updated Task struct:**

```go
type Task struct {
    ID        string
    Name      string
    Cmd       string    // shell command to execute
    State     State
    Priority  int
    DependsOn []string
    CreatedAt time.Time
}
```

This is a shared dependency. To avoid merge conflicts, make it Task 0 — a prerequisite both parallel workers depend on.

## Risks

- **No persistence.** Both commands use MemoryStore, so `add` + `list` in separate invocations won't share state. This is known and expected — persistence comes with BoltDB later. For now, tests validate behavior via injectable stores.
- **Cmd field addition.** Adding a field to Task is backwards-compatible — existing code that doesn't use it is unaffected. MemoryStore stores the full struct.
- **ID collision.** 4 random bytes = 2^32 possibilities. Negligible collision risk for a local tool. Not a concern.
