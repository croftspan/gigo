# Orchestrator Scaffold Generator

Algorithm for scaffolding autonomous execution infrastructure from a GIGO assembly.
Referenced by SKILL.md Step 6.8 when `--include-orchestrator` flag is active.

## Input

Four sources, read in order:

1. Just-completed team assembly — infer project domain from personas and stack
2. `.claude/rules/standards.md` — governance extraction source
3. Project files — test framework and linter detection
4. `http://localhost:8080/v1/models` — local model detection

## Output

Single directory: `vault/` at the project root.

```
vault/
├── _schema/
│   ├── ticket.md
│   └── proof-of-work.md
├── _governance/
│   └── PROJECT_RULES.md
├── _orchestration/
│   ├── hermes-config.yaml
│   ├── dispatch-rules.md
│   ├── escalation-protocol.md
│   └── llama-server.md
├── tickets/
│   └── TEMPLATE.md
├── skills/
│   ├── vault-dispatcher/
│   │   └── SKILL.md
│   ├── proof-of-work/
│   │   └── SKILL.md
│   └── circuit-breaker/
│       └── SKILL.md
├── agents/
│   ├── model/
│   ├── claude-code/
│   └── circuit-breaker/
└── runbooks/
    ├── daemon-setup.md
    ├── model-setup.md
    └── operations.md
```

## Algorithm — Detection

### Step D1: Domain Detection

Map from the just-completed team assembly to a domain category:

| Signal | Domain |
|---|---|
| Personas reference Rails, Django, React, API, database, etc. | `software` |
| Personas reference Roblox, Luau, Studio, TestEZ | `game-roblox` |
| Personas reference Unity, Unreal, Godot, game engines | `game-other` |
| Personas reference fiction, narrative, editing, publishing | `writing` |
| Personas reference papers, methodology, citations, analysis | `research` |
| None of the above | `other` |

### Step D2: Test Framework Detection

Scan project files for framework markers:

| Marker files | Framework |
|---|---|
| `pytest.ini`, `conftest.py`, `tests/` with `.py` files | `pytest` |
| `jest.config.*`, `__tests__/`, `*.test.ts`, `*.test.js` | `jest` |
| `spec/`, `.rspec`, `Gemfile` containing `rspec` | `rspec` |
| `*.spec.luau`, files referencing `TestEZ` | `testez` |
| None found | `null` |

### Step D3: Linter Detection

Scan for linter config:

| Marker files | Linter |
|---|---|
| `.eslintrc*`, `eslint.config.*` | `eslint` |
| `.rubocop.yml` | `rubocop` |
| `selene.toml` | `selene` |
| `.flake8`, `pyproject.toml` with `[tool.ruff]` | `ruff` |
| None found | `null` |

### Step D4: Local Model Detection

Send `GET http://localhost:8080/v1/models`.

- Success: extract `data[0].id` as the model name.
- Failure (connection refused, timeout): use placeholder `"PLACEHOLDER — run llama-server then re-run assembly"`.

Same detection as Brief 07 (`skills/execute/references/local-model-routing.md`, Detection section).

## Algorithm — File Generation

### Step G1: Create Directory Structure

Create all directories listed in the Output section. All `agents/` subdirectories are created empty.

### Step G2: Generate `vault/_schema/ticket.md`

Read `references/ticket-schema.md`. Copy the Frontmatter Schema section. Substitute the detected domain's proof-of-work `required` and `conditional` arrays from the Domain-Aware Proof-of-Work table.

### Step G3: Generate `vault/_schema/proof-of-work.md`

Read `references/ticket-schema.md`. Extract the row matching the detected domain from the Domain-Aware Proof-of-Work table. Write:

```markdown
# Proof-of-Work Schema

## Domain
{detected domain}

## Required Artifacts
{list from domain row — e.g., [test_output, lint_output, reviewer_verdict]}

## Commands
test_command: {detected test framework command or "PLACEHOLDER — configure your test command"}
lint_command: {detected linter command or "PLACEHOLDER — configure your lint command"}

## Conditional Artifacts
{list from domain row, or "None" if empty}
```

### Step G4: Generate `vault/_governance/PROJECT_RULES.md`

Read `.claude/rules/standards.md`. Extract and transform:

1. `## Quality Gates` — copy each bullet as an imperative rule.
2. `## Anti-Patterns` — invert each to a positive imperative. "Codebase overviews in rules" → "Keep codebase overviews out of rules files."
3. `## Forbidden` — copy each bullet verbatim.

Write as a flat rule list:

```markdown
# {Project Name} — Governance Rules

Extracted from .claude/rules/standards.md during assembly.

## Rules
{numbered list of all extracted rules in imperative voice}
```

### Step G5: Generate `vault/_orchestration/hermes-config.yaml`

Use the detected model from Step D4:

```yaml
# Generated by GIGO --include-orchestrator
# Copy to ~/.hermes/config.yaml or merge with existing

model: "{detected_model_id or placeholder}"

providers:
  local:
    api: "http://localhost:8080/v1"
    # context_length is controlled by llama-server -c flag, not this config

agent:
  max_turns: 90
  gateway_timeout: 1800
  tool_use_enforcement: "auto"

terminal:
  backend: local

_config_version: 16
```

### Step G6: Generate `vault/_orchestration/dispatch-rules.md`

Write the vault-dispatcher's operating rules:

```markdown
# Dispatch Rules

## Ticket Selection
Pick the next ticket where `status == "ready"` and all `depends_on` tickets have `status == "done"`.
Ties broken by lowest sequence number in ticket ID.

## Parallelism
Maximum 3 concurrent tickets via Hermes `delegate_task()`.
Check `produced_files` arrays for overlap before parallel dispatch.
Overlapping tickets serialize.

## Retry Budget
1 retry per ticket (same model, error context appended).
After retry failure: escalate to Claude Code.

## Escalation Trigger
2 consecutive failures (retry + original) trigger escalation.
```

### Step G7: Generate `vault/_orchestration/escalation-protocol.md`

Write the Claude Code invocation pattern:

````markdown
# Escalation Protocol

When a ticket fails after retry, invoke Claude Code headless:

```bash
claude -p "{ticket context + prior attempt + error output}" \
  --permission-mode acceptEdits \
  --output-format json \
  --max-turns 25 \
  --allowedTools "Read,Edit,Write,Bash"
```

## After Escalation
1. Parse JSON output for file changes and session ID.
2. Apply file changes to the project.
3. Re-run proof-of-work (test + lint).
4. If pass: set `status: done` with `model: "claude-code"` in history.
5. If fail: set `status: failed`. Log full output to `vault/agents/claude-code/{ticket-id}/`.
````

### Step G8: Generate `vault/_orchestration/llama-server.md`

````markdown
# llama-server Configuration

```bash
llama-server \
  -m ~/models/{model-file}.gguf \
  --host 0.0.0.0 --port 8080 \
  -c 65536 -np 1 -ngl 99 \
  --flash-attn on --jinja
```

## Required Flags
- `--jinja` — required for Hermes tool calling
- `-c 65536` — minimum 64K context required by Hermes

## Model
{detected model path from Step D4, or "Replace with your model path"}
````

### Step G9: Generate `vault/skills/`

Copy three Hermes skill templates from GIGO references to vault:

1. Read `references/hermes-skills/vault-dispatcher.md` → write to `vault/skills/vault-dispatcher/SKILL.md`
2. Read `references/hermes-skills/proof-of-work.md` → write to `vault/skills/proof-of-work/SKILL.md`
3. Read `references/hermes-skills/circuit-breaker.md` → write to `vault/skills/circuit-breaker/SKILL.md`

Copy verbatim. No modifications.

### Step G10: Generate `vault/tickets/TEMPLATE.md`

Write an empty ticket with the full schema from `references/ticket-schema.md`. All fields populated with placeholder values. Include field-level comments explaining each field. This is a template — not a real ticket.

### Step G11: Generate `vault/runbooks/`

**daemon-setup.md:**
```markdown
# Hermes Daemon Setup

1. Install Hermes: `pip install hermes-agent`
2. Copy config: `cp vault/_orchestration/hermes-config.yaml ~/.hermes/config.yaml`
3. Run setup wizard: `hermes setup` (configures gateway platform — Telegram, Discord, etc.)
4. Install custom skills: `cp -r vault/skills/* ~/.hermes/skills/`
5. Start daemon: `hermes gateway install` (creates systemd/launchd service)
   Or foreground: `hermes gateway run`
```

**model-setup.md:**
```markdown
# Local Model Setup

See `vault/_orchestration/llama-server.md` for the startup command.

## Model Download
Download a GGUF model compatible with Hermes tool calling.
Place in `~/models/` and update the `-m` path in llama-server.md.
```

**operations.md:**
```markdown
# Operations

Gateway commands (send via Telegram/Discord):

| Command | Effect |
|---|---|
| `/status` | Current ticket, escalation rate, circuit breaker state |
| `/pause` | Stop dispatching new tickets |
| `/resume` | Restart after pause or circuit breaker trigger |
| `/tickets` | List ready/in_progress/done/failed counts |
```

### Step G12: Create Empty Agent Directories

Create `vault/agents/model/`, `vault/agents/claude-code/`, `vault/agents/circuit-breaker/`. These are populated during execution.
